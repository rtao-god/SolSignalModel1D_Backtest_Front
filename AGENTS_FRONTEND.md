## Frontend-специфика
- Не меняй архитектуру фронтенда, UI-контракты и API-контракты без явного запроса.
- В `scss/css` по умолчанию используй `px` для шрифтов, отступов, мелкой геометрии и точных размеров; `rem` оставляй только для крупных контейнеров, крупных ограничений ширины и похожих размеров, где действительно нужна привязка к базовому масштабу страницы.
- Сайт сейчас по умолчанию живёт в тёмной теме; новые стили для компонентов подбирай с учётом тёмного фона как базового сценария и заранее проверяй контраст текста, границ, теней и состояний наведения/фокуса именно на нём.
- `TermTooltip` (`src/shared/ui/TermTooltip/ui/TermTooltip.tsx`) обязателен для терминов, доменных сокращений и потенциально непонятного сложного поведения в UI.
- Не добавляй `TermTooltip` к бытовым очевидным словам интерфейса (например: `кнопка`, `таблица`, `день`), если они не несут доменной нагрузки.
- Для доменных терминов отчёта/риска/режимов (`policy`, `liquidation`, `drawdown`, `bucket`, `tp/sl`, внутренние фильтры) `TermTooltip` обязателен, если без него возможна неверная интерпретация чисел и поведения стратегии.
- Для каждого доменного термина в пользовательском тексте проверяй наличие tooltip: если термин может быть непонятен без контекста, он должен быть обёрнут в `TermTooltip`.
- После любой правки user-facing текста перечитывай именно финальный вариант строки или абзаца и выписывай все доменные термины в нём. Каждый такой термин должен быть либо обёрнут в `TermTooltip` / `[[term|label]]`, либо переписан в более простой недоменный язык без потери смысла.
- Отдельно проверяй термины, которые легко пропустить как "почти обычные" слова: `плечо`, `стартовый капитал`, `размер позиции`, `выведенная прибыль`, `реинвестирование`, `спотовая стратегия`, `журнал реальных прогнозов`, `фильтры`, `риск-слои`, `SL` / `TP`. Если такой термин остался plain text без tooltip, правка считается незавершённой.
- Для технических терминов и индикаторов пиши каноничное имя прямо в тексте (например, `ATR индикатор`, `SMA 50`, `EMA 200`) и используй единое описание этого термина во всех местах UI.
- Один и тот же термин во всех tooltip и связанных текстах называй одинаково; не переименовывай один смысл разными формулировками и не переписывай словарное определение заново в каждом месте.
- Перед добавлением нового доменного термина или alias сначала проверь `src/shared/terms/**` и уже существующие aliases этого смысла.
- Если такой смысл уже есть под другим названием, переиспользуй существующий canonical term id и его описание, а не создавай новую формулировку рядом.
- Локальный термин в `pages/*` или другом page-level файле допустим только если он реально нужен одной странице и не дублирует общий термин из `src/shared/terms/common/**` или report-specific каталогов.

## Подробные ошибки и логи (обязательно)
- Любая frontend-ошибка, предупреждение, runtime-log или error boundary message, который видит разработчик, оператор или пользователь, должен объяснять: что ожидалось, что было найдено, где искали, какой owner-контракт сломался и что делать дальше.
- Для frontend-ошибок и логов это означает обязательные поля `owner`, `expected`, `actual`, `requiredAction`, `context` или эквивалентный по смыслу набор; без этого сообщение считается недостаточным.
- Генерировать голый `Error`, пустой лог, безымянный `unexpected error` или скрытую поломку без owner-контекста и следующего действия запрещено.
- Если ошибка или лог проходит через общий helper, boundary, mapper или logger, этот слой обязан сохранить исходный контекст, а не сводить причину к общей фразе.
- Если подробное сообщение невозможно без дополнительного кода, сначала добавить контекст, а уже потом показывать ошибку или писать лог.
- Это правило относится и к техническим ошибкам, и к debug/diagnostic логам, и к сообщениям, которые потом попадают в UI-сборку или консоль.

## User-facing тексты (frontend)
- Это правило действует для всех пользовательских текстов фронта: `title`, `subtitle`, `description`, `hint`, навигационных пояснений, onboarding-текстов, explain-блоков и других product-facing описаний.
- Это правило не действует на явно developer-only материалы и на чисто служебные технические ошибки, где нужен точный operational/debugging контекст.
- Пользовательские тексты на фронте пишутся строго нейтрально. Запрещены любые обращения к читателю и командные формулировки: `ты`, `вы`, `смотри`, `перейдите`, `выберите`, `нажмите`, `откройте`, `тыкните`, `здесь можно посмотреть` и аналогичные конструкции.
- Вместо команд использовать описательные формулировки: `страница показывает`, `раздел содержит`, `здесь отображается`, `в этом блоке виден`, `раздел нужен для`, `на этой странице собраны`.
- Базовая целевая аудитория user-facing текстов на фронте:
  - CTO компании;
  - предприниматель;
  - системный аналитик;
  - инвестор.
- По умолчанию текст должен подходить всей этой аудитории даже без крипто-контекста, без контекста моделирования и без знания механики бэктеста.
- Если текст понятен только человеку, который уже знает крипту, ML или механику бэктеста, он не проходит по умолчанию для этой аудитории и должен быть переписан.
- Любое доменное действие доводить до логического объекта, если без него фраза двусмысленна для читателя без контекста: `вход` -> `вход в сделку`, `выход` -> `выход из сделки`, `сигнал` -> `сигнал на вход в сделку` или другой точный объект по месту.
- Примеры формулировок:
  - Плохо: `Откройте диагностику и посмотрите, почему стратегия ошиблась.` Хорошо: `Диагностика показывает, на каком шаге стратегия потеряла качество и какой блок дал слабый результат.`
  - Плохо: `Сигнал не подтвердился.` Хорошо: `Сигнал на вход в сделку не подтвердился: метрики качества входа остались ниже порога.`
- Перед финализацией любого user-facing текста делать self-check:
  - нет ли обращения к читателю и скрытых императивов;
  - понятен ли текст без крипто-контекста;
  - не требует ли текст знания ML/бэктеста для базового понимания;
  - все неизбежные доменные термины либо раскрыты на уровне смысла в основном тексте, либо вынесены в tooltip;
  - при повторном чтении финального текста не осталось ни одного доменного термина plain text без tooltip; если такой термин остался, текст считается незавершённым;
  - отдельно проверены "ложно обычные" термины: `плечо`, `стартовый капитал`, `размер позиции`, `выведенная прибыль`, `реинвестирование`, `спотовая стратегия`, `журнал реальных прогнозов`, `фильтры`, `риск-слои`, `SL` / `TP`;
  - нет ли обрезанных доменных фраз, после которых у читателя без контекста остаётся вопрос `что именно имеется в виду`: `вход куда`, `выход из чего`, `сигнал на что`, `риск чего`;
  - подходит ли текст одновременно для CTO, предпринимателя, системного аналитика и инвестора, даже если у них нет крипто-контекста, контекста моделирования и знания механики бэктеста; если нет, текст переписывается.

### Стиль письма для white paper и пользовательских описаний
- Этот блок обязателен для help-текстов, explain-блоков, документационных страниц, white paper-style описаний и длинных пользовательских пояснений на фронте.
- Для этого блока обязателен глобальный порядок из `.codex/Rules/WRITING_RULES.md` -> `### Протокол написания пользовательского и документационного текста (обязательно)`; менять этот порядок местами запрещено.
- Писать строго для человека без технического контекста.
- Каждая строка должна быть понятна сама по себе без знания кода, файлов, API и внутренней терминологии проекта.
- Любой термин при первом упоминании раскрывать до полного бытового смысла.
- Не писать `Solana`, если по смыслу имеется в виду цена актива; писать `цена монеты Solana` или другой полный объект по месту. Но если в тексте уже был полный термин, то дальше можно писать чуть укороченную версию по типу `цена Sol`.
- Не использовать в пользовательском тексте имена классов, DTO, файлов, путей, ключей, внутренних режимов хранения и других сущностей реализации.
- Если мысль можно сказать прямо, писать прямо; не прятать нормальную фразу в скобки.
- Скобки использовать только тогда, когда они добавляют короткий и полезный второй слой смысла, а не дублируют основную фразу.
- Предпочитать конкретику общим словам: точные годы, окна, проценты, режимы и границы вместо расплывчатых формулировок.
- Если строку можно усилить одним коротким и реально полезным уточнением, это уточнение нужно добавить.
- Не писать фон, который не объясняет сам продукт: историю рынка, общие рассуждения про крипту, маркетинговые вступления и другой шум.
- Не писать бесполезные внутренние ограничения, которые не меняют понимание продукта пользователем.
- Оставлять только те ограничения, которые реально влияют на чтение результата: что учитывается, что не учитывается, где история может пересчитываться, как это влияет на интерпретацию.
- Перед финализацией делать буквальный построчный аудит: каждая строка должна либо добавлять понятный смысл, либо удаляться.
- Построчный аудит пользовательского текста делать по доменной пользе, а не по связности. Строка, которая описывает сам документ, ход работы агента, редакторскую подводку или просто склеивает соседние мысли без нового знания о продукте, считается бредом и подлежит удалению или переписыванию.
- В этом аудите отдельно искать места, где читатель споткнётся: перечитает строку, будет догадываться о связи слов или спросит `что это`, `о чём это`, `зачем это здесь`. Любое такое место считается дефектом даже при гладкой грамматике и должно быть переписано или удалено.
- Если после удаления строки понимание не ухудшается, строка лишняя.
- Делай не редактуру, а враждебный редакторский аудит.
- Смотри на текст как человек, который хочет найти максимум придирок.
- Для каждой строки сначала докажи, зачем она нужна читателю.
- Если строка не добавляет новый смысл, удаляй.
- Если есть формальная подводка, слабое слово, лишняя частица `не`, повтор, недораскрытый термин, факт без объяснения `зачем это знать`, технический жаргон или вода — переписывай.
- После правки сам найди минимум 20 придирок к своему же тексту и устрани их до финала.
- Если user-facing текст содержит 2+ самостоятельные мысли, причины, эффекта, шага чтения или примера, по умолчанию оформляй его списком, а не сплошным полотном текста.
- Для нумерованной последовательности, приоритета, этапов, порядка чтения и причинно-следственной цепочки используй формат `1)`, `2)`, `3)`.
- Для набора равноправных тезисов без жёсткого порядка используй `BulletList` (`src/shared/ui/BulletList/ui/BulletList.tsx`) или эквивалентный shared-список с тем же контрактом, а не стандартный `li`-маркер по умолчанию.
- Выбор между `1)` и `BulletList` делать по смыслу, а не по вкусу: если порядок важен — нумерация, если важна просто группировка тезисов — bullet-список.
- Основной блок правил написания tooltip:
  - tooltip должен объяснять смысл термина или поведения, а не быть короткой формальной подписью;
  - для сложного поведения обязательно раскрывать: что это, почему это происходит, что из-за этого меняется в числах/поведении, как это читать пользователю;
  - в основном тексте не дублировать словарное определение термина, если оно уже вынесено в tooltip;
  - если без примера пользователь не поймёт интерпретацию метрики, режима или условия, добавляй короткий предметный пример;
  - длинные tooltip разбивай на смысловые абзацы с пустой строкой между блоками; под `красной строкой` в этих правилах понимать именно отдельный абзац, а не CSS-отступ;
  - красную строку используй для отделения новой идеи, новой причины, нового условия, нового шага чтения или нового примера; tooltip не должен выглядеть сплошным полотном, в тексте нужен воздух и явное деление по смыслу;
  - если в tooltip есть несколько независимых причин, условий, шагов чтения или правил интерпретации, оформляй их как `1)`, `2)`, `3)`, а не сплошным полотном;
  - в описании tooltip запрещена самоссылка на этот же термин: внутри tooltip для `Bucket` не должно быть кликабельного `Bucket`, внутри tooltip для `Policy` — кликабельного `Policy` и т.д.; при rich-text рендере используй `excludeTerms`.
  - перед финализацией tooltip с rich-text разметкой `[[term|label]]` обязательно делать preflight на два дефекта: 1) нет ли самоссылки на текущий термин по любому его представлению (`raw key`, локализованный title, canonical term id/label); 2) нет ли битой разметки вида `[[term|label]`, `[[term|label` или другого сырого explicit-term синтаксиса, который утечёт в UI текстом.
- Стиль подсказок формата `Почему?`: отвечать именно на причину (`почему это происходит`), а не на назначение (`зачем это сделали`).
- Для `Почему?`-подсказок используй причинно-следственную структуру: рыночная/модельная причина -> что из-за этого меняется в данных или поведении стратегии -> как это влияет на интерпретацию метрик.
- В `Почему?`-подсказках запрещены формулировки «потому что так настроено», «в этой конфигурации не используется» без раскрытия фактической причины.
- Единые правила комментариев находятся в `.codex/Rules/COMMENT_AND_NAMING_RULES.md`.
- Для frontend также обязательны подпункты про совместное использование `summary` + `//`, ограничения на `<c>...</c>` и краткую расшифровку неочевидных терминов.
- Для frontend отдельно обязателен антишумовой гейт из `.codex/Rules/COMMENT_AND_NAMING_RULES.md`: запрещены исторические, риторические и шаблонные комментарии; комментарий описывает только текущую UI-логику, инвариант или ограничение.
- Для frontend применяй эти правила к `SolSignalModel1D_Backtest_Front/**`; комментарии обязательны не только в сложных, но и в средних по сложности местах, если они реально сокращают время входа в файл новому разработчику без контекста фронтенда и бэкенда.
- В каждом нетривиальном frontend-файле должен быть хотя бы один реально полезный комментарий: либо главный комментарий файла, либо локальный комментарий у важного контракта, ограничения, ветвления, UI-механики, состояния, синхронизации или причины решения.
- Цель комментариев во frontend-коде: быстро объяснить что делает файл, зачем он нужен, откуда приходят данные, какие ограничения и инварианты важны, и что в логике нельзя читать буквально без знания проекта.
- Если файл не является совсем тривиальной обёрткой или реэкспортом, молчаливый код без единого полезного комментария считается дефектом оформления.
- Комментарий должен быть полезным, а не формальным: он должен экономить время чтения новому разработчику, а не повторять очевидные имена переменных, пропсов и JSX-структуры.
- В сложной и средней UI-логике используй главный комментарий файла вместе с локальными `//` пояснениями по инвариантам, ограничениям, источникам данных и причинам нетривиального поведения.
- Build/lint запускать целево: когда изменения реально затрагивают сборку, линт-правила или runtime-поведение UI.
- Для чисто документационных/организационных правок не запускать build/lint по умолчанию.
- Если проверки запускались, по возможности используй:
  - `npm run build`
  - `npm run lint`
- Если проверки не запускались или завершились с ошибкой, явно укажи это в отчёте.

- Запрещены абстрактные формулировки без механики (`включает`, `учитывает`, `обычная логика`, `по условиям стратегии`, `может влиять`); вместо них писать конкретный триггер и конкретный эффект в метриках.
- Для режимов выхода из сделки всегда явно перечислять доступные события закрытия: `TP` / `SL` / `ликвидация` / `EndOfDay` (или точный поднабор для конкретного режима).
- Для каждого доменного слова внутри tooltip обязателен вложенный tooltip, кроме самоссылок текущего термина и очевидных общеязыковых слов.
- Перед финальной формулировкой проверять чек-лист:
  - читатель без крипто-контекста может пересказать механику режима;
  - ясно, какие метрики изменятся при переключении;
  - есть короткий пример чтения;
  - нет доменных терминов без tooltip.

## Системные находки и гейт перед финалом (обязательно)
- Для задач с изменением UI-логики, рендера, tooltip или контрактов отображения в финальном отчёте обязателен блок `Системные находки`:
  - какая корневая проблема найдена;
  - где ещё в проекте проявляется тот же паттерн;
  - что исправлено в текущем диффе;
  - что осталось и почему это не закрыто сейчас.
- Гейт перед финалом:
  - если найден повторяющийся паттерн дефекта, задачу нельзя закрывать точечной правкой одного места;
  - нужно либо выполнить системный проход по всем релевантным точкам, либо явно остановиться с перечнем оставшихся мест и причиной остановки.
- Для повторяющихся дефектов в рендере/tooltip обязательно фиксировать критерий поиска скоупа (поиск по коду, контрактный список, карта компонентов), чтобы покрытие исправления было проверяемым.
